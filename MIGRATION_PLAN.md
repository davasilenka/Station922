# План миграции Station922 CGI к полному соответствию task.md

Этот документ для тебя (владельца проекта). Он содержит:
- общий план разбит на 2 основные сессии + 1 резервную
- готовые промпты для копирования в начале каждой сессии
- сигналы, по которым понять идёт ли работа хорошо или пора останавливать

## Архитектура плана

Цель — закрыть все расхождения из [TASK_COMPLIANCE.md](TASK_COMPLIANCE.md), не теряя 32/32 теста. Разбит на этапы по принципу:
- **Фаза 1 — низкий риск**: удалить мёртвый код, добавить недостающие фичи (stderr, allowed dirs), перейти на named pipes БЕЗ изменения семантики I/O.
- **Фаза 2 — высокий риск**: включить `FILE_FLAG_OVERLAPPED` + IOCP, переписать I/O на async через `IBaseAsyncStream`. Это **самая сложная часть** (на ней застревали предыдущие агенты).
- **Фаза 3 (резерв)**: если фаза 2 не завершилась за одну сессию — продолжение с того места.

После каждой фазы — коммит + push + 32/32 тестов. Если фаза не доходит до 32/32 — её результаты НЕ коммитим, откатываемся к предыдущему коммиту, и идём в фазу-резерв.

## Фаза 2 — Async I/O через IOCP (главный этап)

**Что закрываем:**
- `FILE_FLAG_OVERLAPPED` на server-end pipes
- Привязка pipe handles к IOCP `ThreadPool->CompletionPort`
- Новый модуль `PipeAsyncStream.bas`, реализующий `IBaseAsyncStream` поверх pipe (по аналогии с [NetworkAsyncStream.bas](src/NetworkAsyncStream.bas))
- Переписать Step 9 (stdin write) и Step 10 (stdout read) в `HttpCgiProcessor.bas` на `BeginRead/BeginWrite`
- Удалить sync `PeekNamedPipe`+`Sleep(10)` цикл
- Stderr drain тоже через `PipeAsyncStream`

**Ожидаемый объём:** ~400-600 строк. Риск регрессии — ВЫСОКИЙ. Это место где предыдущие агенты ломались.

**Промпт для копирования агенту в начале сессии:**

```
Прочитай ТОЛЬКО эти файлы в указанном порядке, прежде чем что-либо делать:
1. session_handoff_phase2.md — твой план работы на эту сессию
2. task.md — единственный источник истины по требованиям
3. src/NetworkAsyncStream.bas — образец реализации IBaseAsyncStream через IOCP

После прочтения сделай:
1. Запусти test_cgi.py и убедись что отправная точка — 32/32. Если не 32/32 — НЕ начинай работу, доложи и жди указаний.
2. Покажи мне детальный план КАЖДОГО подэтапа из session_handoff_phase2.md (особенно lifecycle named pipe — connect/inherit/close). Жди подтверждения по каждому подэтапу отдельно.

Главные правила:
- Это технически сложная фаза. ПОСЛЕ КАЖДОГО ПОДЭТАПА — прогон test_cgi.py. Если тесты не 32/32 — откатывай этот подэтап и доложи мне, не пытайся починить сложными правками.
- Маленькие коммиты + git push origin master после каждого подэтапа.
- Если за 3 попытки подэтап не сходится — стоп, доложи мне симптомы (логи, ошибки, какие тесты падают), не продолжай.
- Файлы .bas/.bi в Windows-1251 — в Edit/Write пиши только ASCII в новых строках.
- Не трогай интерфейсы IBaseAsyncStream, IAsyncResult — только добавляй НОВЫЙ модуль PipeAsyncStream который их РЕАЛИЗУЕТ.

Особые предупреждения по граблям (детали в session_handoff_phase2.md):
- Анонимные pipes (CreatePipe) НЕ поддерживают OVERLAPPED. Только named pipes (CreateNamedPipeW). Если ты пытаешься сделать overlapped на CreatePipe — остановись, это не работает.
- ConnectNamedPipe (overlapped) обязателен ДО CreateProcess, иначе child повисит на стartup
- CancelIoEx — НЕ освобождать OVERLAPPED-структуру сразу, нужно дождаться completion с ERROR_OPERATION_ABORTED
- Process-pipe race: данные в pipe могут оставаться ПОСЛЕ exit процесса, drain должен быть до broken_pipe, не до exit code
```

**Сигналы прогресса:**
- ✓ Хорошо: «подэтап N (stdin async): создал PipeAsyncStream, переписал Step 9, тесты 32/32, push»
- ⚠ Внимание: «тест slow.py timeout висит» — это race с CancelIoEx, не пускай агента «допиливать» долго
- ⚠ Внимание: «тест 5 parallel падает на 1-2 запросах» — handle inheritance / named pipe naming race, агент должен показать тебе план фикса
- ✗ Стоп: «сервер падает моментально на любом CGI» — откат, переход к фазе 3 с диагностикой

**Критерий завершения фазы 2:**
- 4-6 коммитов на master (по одному на подэтап) + push
- `_dbg_*` файлы убраны
- Sync `PeekNamedPipe`+`Sleep` цикл удалён из HttpCgiProcessor
- TASK_COMPLIANCE.md обновлён — все расхождения закрыты
- Тесты 32/32 (рекомендую 3 прогона подряд для проверки стабильности)

## Фаза 3 — Резерв (если фаза 2 не завершилась)

Используется если фаза 2 ушла в красное и пришлось откатывать. Цель — добить недоделанное с фокусом на одну оставшуюся проблему.

**Промпт для копирования агенту в начале сессии:**

```
Прочитай:
1. session_handoff_phase3.md — что осталось доделать
2. Последний коммит на master (git log -1) — где мы остановились
3. Файлы _dbg_* в корне (если есть) — диагностика от прошлой сессии

После прочтения:
1. Запусти test_cgi.py чтобы понять текущее состояние.
2. Доложи мне: какие тесты падают, какие проходят, и твою гипотезу о причине.
3. Жди моего ответа перед началом работы.

Стратегия фазы 3 — точечный фикс одной проблемы, не широкая переделка. Если тебе нужно больше чем одно изменение чтобы починить — стоп, доложи план.
```

## Итоговая таблица для тебя

| Фаза | Длительность | Риск | Что получаем |
|---|---|---|---|
| 1 — безопасные доработки | 1 сессия (~2-4 часа агентом) | низкий | stderr, allowed dirs, named pipes; 32/32 |
| 2 — async через IOCP | 1 сессия (~3-6 часов агентом) | **высокий** | полное соответствие task.md; 32/32 |
| 3 — резерв | 0-1 сессия | переменный | добивание оставшейся проблемы |

**Реалистичный прогноз** (от меня, агента который видел эти грабли вживую):
- Шанс уложиться в 2 сессии: ~55%
- Шанс что потребуется фаза 3: ~30%
- Шанс что придётся откатываться к текущему master и принимать архитектурные расхождения: ~15%

**Что повышает шансы:**
- Маленькие коммиты с тестом после каждого
- Жёсткие stop conditions для агента (если не сходится за 3 попытки — стоп)
- Не давать агенту «допиливать» проблему дольше 30 минут на одно место
- Перечитывать handoff каждой фазы перед стартом — там грабли описаны

**Что снижает шансы:**
- Если агент начинает менять интерфейсы (IBaseAsyncStream, IAsyncResult)
- Если агент пытается сделать всё в одном большом коммите
- Если агент не убирает диагностические `_dbg_*` файлы перед финальным коммитом
