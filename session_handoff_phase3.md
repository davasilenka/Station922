# Handoff — Фаза 3 (резерв): Добивание после неудачной фазы 2

Эта сессия запускается **только если фаза 2 не довела проект до 32/32** или была частично завершена. Цель — точечный фикс или возврат к рабочему состоянию.

## Первое что сделать

1. Посмотри `git log -15 --oneline` — где остановилась фаза 2.
2. Запусти `test_cgi.py` (сначала убедившись что сервер собирается).
3. Сравни последний коммит фазы 2 с `d35997f` (точка после фазы 0 — гарантированно работающее состояние).

## Возможные сценарии

### Сценарий А — несколько тестов падают

Например, 28/32, падают `5 parallel`, `POST body`, `slow.py timeout`. Это типичная картина проблем фазы 2.

**Стратегия:** диагностика одного конкретного теста, не широкий рефакторинг.

1. Выбери ОДИН падающий тест (рекомендую сначала самый простой — например, `POST body` если он падает; `5 parallel` сложнее).
2. Добавь точечную диагностику в код (по образцу диагностики из коммита `d35997f` — там были `_dbg_post.txt`, `_dbg_fail.txt`).
3. Запусти тест, посмотри что пишется в `_dbg_*`.
4. Сформулируй гипотезу о причине.
5. Доложи мне и жди указаний — не пытайся починить большим патчем.

### Сценарий Б — сервер падает на первом же CGI

Это худший случай. Что-то фундаментально сломано в pipe lifecycle или CreateProcess.

1. Сравни `src/CgiProcess.bas` с версией из коммита `d35997f`:
   ```
   git diff d35997f -- src/CgiProcess.bas
   ```
2. Сравни `src/HttpCgiProcessor.bas`:
   ```
   git diff d35997f -- src/HttpCgiProcessor.bas
   ```
3. Доложи пользователю что изменилось и попроси выбрать стратегию:
   - **Откатиться к `d35997f`** (потеря работы фазы 2, но 32/32 гарантированы)
   - **Точечный фикс** (если ты видишь конкретную проблему — например, забыли вызвать `ConnectNamedPipe`)

### Сценарий В — фаза 2 завершилась с 32/32, но не все подэтапы сделаны

Например, Step 9 (stdin) переведён на async, а Step 10 (stdout) ещё нет.

1. Продолжи с того подэтапа, на котором остановилась фаза 2.
2. Следуй инструкциям из [session_handoff_phase2.md](session_handoff_phase2.md), но только для оставшихся подэтапов.

## Откат как опция

В любой момент можно откатиться к гарантированно рабочему состоянию:

```powershell
git reset --hard d35997f
git push origin master --force-with-lease
```

**Force-push требует подтверждения пользователя!** Это переписывание истории.

Альтернатива без force-push — создать revert-коммиты:

```powershell
git revert <hash_первого_коммита_фазы_2>..HEAD --no-edit
git push origin master
```

Это сохранит историю и вернёт код к состоянию `d35997f`.

## Stop conditions

- Если диагностика не даёт зацепку за 3 попытки — стоп, доложить пользователю.
- Если предложенный фикс не работает за 2 итерации — стоп.
- Если появляется новая регрессия (тест, который ещё проходил, перестал) — немедленный откат изменения.

## По завершении сессии

Возможные исходы:
1. **Доделали до 32/32, все подэтапы фазы 2 закрыты** — обновить TASK_COMPLIANCE.md, push, доложить пользователю «полное соответствие task.md достигнуто».
2. **Доделали до 32/32, но часть подэтапов фазы 2 пропущена** — TASK_COMPLIANCE.md показывает частичное соответствие, доложить пользователю что осталось.
3. **Откатились к `d35997f`** — доложить пользователю, обновить TASK_COMPLIANCE.md (с архитектурными расхождениями), показать преподавателю.

## Правила работы

- Те же что и в фазах 1-2.
- Дополнительно: **минимум изменений**. Это сессия фикса, не рефакторинга.

## Ключевые файлы

- [task.md](task.md) — требования
- [TASK_COMPLIANCE.md](TASK_COMPLIANCE.md) — статус соответствия
- Последние `_dbg_*.txt` (если есть) — диагностика от фазы 2
- `git log -20` и `git diff d35997f` — путь от рабочего состояния к текущему
